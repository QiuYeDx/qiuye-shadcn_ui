# 快速上手页面重构开发设计文档

## 背景

旧 `/cli` 页面以“CLI 工具”为中心组织内容，包含安装、使用、命令、API、注册表结构等多段说明。页面信息密度高但目标混杂，容易让首次访问的使用者误以为需要理解一个独立 CLI 产品，而不是快速把 QiuYe UI 组件接入自己的 shadcn/ui 项目。

本次重构将旧页面彻底替换为面向使用者的“快速开始”页面。保留 `/cli` 路由用于兼容已有链接，但页面标题、导航文案和内容结构都应转向“如何开始使用本项目”。

## 目标

- 将 `/cli` 改造成清晰、专业、可操作的快速上手页面。
- 首屏直接说明 QiuYe UI 的使用路径：初始化 shadcn/ui、配置 QiuYe UI registry、添加组件、查看组件。
- 保留 npm / pnpm 命令切换和一键复制能力。
- 用真实组件数据展示可安装组件入口，避免硬编码旧组件清单。
- 更新站点导航文案，降低“CLI 工具”感。
- 保持现有 registry、组件详情页和组件列表页不受影响。

## 非目标

- 不修改组件 registry 生成逻辑。
- 不新增后端接口或动态数据源。
- 不改 `/cli` 路由路径，避免破坏外部链接。
- 不重写组件详情页、首页和组件列表页。
- 不引入新的 UI 依赖。

## 重构前状态

- `app/cli/page.tsx` 是旧 CLI 页面，整页为客户端组件，依赖 `motion/react`、`next-themes`、`use-clipboard-copy`、`ResponsiveTabs`、`CodeBlock`、`SmoothCorners` 等。
- 旧页面以四个 tab 组织：安装、使用、命令、API。内容里同时包含 shadcn init、registry 配置、直接 URL 安装、API 结构示例和组件清单，信息层级不稳定。
- `components/site/site-nav.ts` 桌面与移动主导航中展示 `CLI`。
- `components/site/site-header.tsx` 右侧主按钮显示 `Install` 并指向 `/cli`。
- `components/app-sidebar.tsx` 是旧布局遗留入口，重构前展示 `CLI 工具`，当前 `SiteShell` 不直接使用它，但保留更新以避免未来引用时文案不一致。
- `components/header.tsx` 是旧 Header 组件，当前布局不直接使用，但重构前同样存在 `CLI 工具` 文案。
- `lib/registry.ts` 提供 `getAllComponents()`，可作为快速开始页的组件数据来源。
- `pnpm-lock.yaml` 为 lockfile v6，应按 pnpm 8 系列校验，避免用新版 pnpm 改写锁文件。

## 实现后状态

- `app/cli/page.tsx` 已替换为“QiuYe UI 快速开始”页面，仍保留 `/cli` 路由。
- `components/site/site-nav.ts` 将 `/cli` 导航展示为 `Quick Start`。
- `components/site/site-header.tsx` 将右侧主按钮展示为 `Start`。
- `components/home/home-hero.tsx` 和 `components/home/home-cta.tsx` 的 `/cli` 入口展示为“快速开始”。
- `components/app-sidebar.tsx` 和 `components/header.tsx` 的遗留入口展示为“快速开始”。

## 用户面对的最终行为

- 用户访问 `/cli` 后看到“快速开始”页面。
- 首屏包含：
  - QiuYe UI 品牌与简短定位。
  - 包管理器切换。
  - “浏览组件”和 GitHub 两个明确行动按钮。
  - 一条可复制的推荐安装命令，作为主要安装动作。
- 页面主体包含：
  - 4 个步骤：准备 shadcn/ui、配置 registry、添加组件、在代码中使用。
  - 中国大陆镜像地址提示，作为接入细节而不是单独文档章节。
  - 常用组件速览，数据来自 registry。
  - 可选的 MCP / AI 编辑器辅助说明，放在次级区域。
  - 简短排错清单。

## 前端设计

- 页面继续使用客户端组件，以保留包管理器切换、复制和 toast 反馈。
- 去掉旧 tab 文档结构，改成纵向页面信息流，降低学习成本。
- 视觉风格采用官网现有设计语言：`container`、`Button`、`Badge`、`SmoothCorners`、`CodeBlockPanel`。
- 卡片仅用于独立信息块，不做嵌套卡片。
- 按钮使用 lucide 图标；复制图标按钮提供 `aria-label` 和 tooltip。
- 布局需在移动端自然换行，命令块不撑破页面。
- 不使用大面积单一色系渐变和装饰性漂浮元素。

## 数据和接口

- 页面不新增网络请求。
- 可安装组件来自 `getAllComponents()`。
- registry 配置沿用：

```json
{
  "registries": {
    "@qiuye-ui": "https://ui.qiuyedx.com/registry/{name}.json"
  }
}
```

- 国际域名仍可在页面中作为备用地址出现：

```text
https://qiuye-ui.vercel.app/registry/{name}.json
```

## 兼容性

- `/cli` 路由保留。
- 旧的 `qiuye-ui-cli` 包、MCP README 和 registry JSON 均不改动。
- 命令生成继续支持 `npx` 与 `pnpm dlx`，默认使用 `pnpm`。

## 风险和边界

- shadcn CLI 的 registry 配置格式未来可能变化；本次按仓库现有页面和 registry 文件保持一致。
- 页面文本应避免承诺“无需配置即可所有场景可用”，因为用户项目仍需要正确初始化 shadcn/ui。
- 组件数据来自 `lib/registry.ts`，若 registry 元数据缺失，页面只展示已有字段，不额外推断。

## 验证策略

- 运行 ESLint，确认无类型层面的明显问题。
- 在可行时运行 Next build，确认页面可编译。
- 如果启动本地前端服务做视觉检查，结束前必须停止该进程。
- 校验命令应使用 pnpm 8 系列或不触碰 lockfile 的命令。
